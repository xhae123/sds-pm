0. 수업 전체의 큰 그림

📌 한 문장 메시지 (이거 하나는 무조건 외울 것) "좋은 프로젝트는 가장 화려한 모델을 쓰는 프로젝트가 아니라, 주제·데이터·모델이 논리적으로 맞물린 프로젝트다."

8개 과제는 모두 이 한 문장의 다른 각도다. 매 지시서의 마지막 슬라이드에 같은 메시지의 변형이 나오고, 모든 평가 기준이 결국 이걸 묻는다. 시험에서 무엇을 묻든, 답의 마지막 한 줄은 이 메시지로 닫을 수 있어야 한다.

1.4 V1/V2 분리의 출처 (가장 중요한 한 줄)

✅ 지시서 원문 (직접 인용 가능) "최종 프로젝트에 반드시 머신러닝이 포함되어야만 하는 것은 아닙니다. Version 1: 머신러닝 없이 작동하는 기본 어플리케이션 / Version 2: 머신러닝 요소가 추가된 확장 버전. ML이 최종 결과물에 포함되지 않더라도 다른 부분이 충실하게 구현되어 있다면 감점하지 않겠습니다."
💡 The Why 이 한 줄이 왜 중요한가? "학기 안에 구현 가능"이 우선이라는 수업 철학의 출처다. ML이 강제되면 학생들은 화려한 모델로 도망치게 되고, Product Goal과 무관한 프로젝트가 만들어진다. V1/V2 분리는 "단순한 V1으로 학기 안에 검증 → 데이터 쌓이면 V2 ML로 확장"이라는 실제 산업 현장의 점진적 접근을 학생에게 가르치려는 의도.

1.5 작성 양식 — 기획안 5항목

항목적어야 할 내용후속 과제 연결
1) 해결하려는 문제어떤 현실 문제 / 누가, 어떤 상황→ 4 (사용자 데이터)
2) 만들고 싶은 어플리케이션어떤 기능 / 사용자 사용 방식→ 8 (UX)
3) 문제 해결 절차 (Procedure)1·2·3단계로 입력→내부절차→결과→ 7 (heuristic)
4) 머신러닝 요소어느 단계에 ML / 어느 단계는 절차적 코딩→ 3 (Model Metric)
5) 전문가 벤치마킹누가 잘 푸는가 / 그 전문가의 판단 기준→ 7 (도메인 전문가)

📌 R-O-C-O (시험에 단답 출제 가능)
약어의미왜 필요한가
Role역할 — AI에게 누구로 답하게관점이 좁혀져야 답이 구체적

1.7 AI 활용 3가지 주의

  1. AI가 준 답에 내 아이디어가 끌려가면 안 된다 — AI 답이 그럴듯해도 우리 의도와 다를 수 있다.
  2. 프롬프트에 맥락을 충분히 넣어야 한다 — R-O-C-O.
  3. 일반적인 앱이 아니라 우리 팀만의 포인트가 필요하다 — 우리 팀이 잘 아는 분야 / 우리만 볼 수 있는 문제.
⚠️ Common Misconceptions
오해 1: "AI가 잘 답해주니 그대로 제출하면 된다." → 틀림. 수업 방침: AI 초안은 검토 대상이지 답이 아니다. 비판적 검토 단계가 평가의 핵심.
오해 2: "ML이 없으면 점수가 깎인다." → 틀림. 지시서 명시 — V1만으로도 감점 안 함.
오해 3: "주제는 거창하고 새로워야 한다." → 틀림. 가까이·익숙한 문제가 더 좋은 출발점이다.

2 Business Metric 도출 (실습 1)

지시서: AI_lecture.pdf 슬라이드 4·5 · "AI 초안 → 팀 비판적 검토" 워크플로우 안에서 진행

2.1 묻는 것

"우리 팀의 제품/기능 목표에 맞는 Business Metric을 도출하라."

2.2 왜 이 과제가 필요한가

💡 The Why 모델 정확도가 95%여도 사용자가 클릭 안 하면 무가치다. "성공을 무엇으로 잴 것인가"가 정해지지 않으면 후속 모델·실험·개선이 모두 방향성을 잃는다. BM은 "우리가 사용자에게 약속한 가치가 실제로 전달됐는지"를 잰다. 즉, BM은 모델 만들기 전에 정해야 한다 (모델 만들고 나서 BM 정하면 자기 모델에 유리한 BM을 고르게 됨).

2.3 핵심 개념 — Business Metric이란?

📌 정의 Business Metric (BM) = 제품·기능이 사용자에게 가치를 제공하는지를 사용자의 행동·결과로 측정하는 지표. 모델 정확도가 아니라 "사용자가 원하는 결과를 얻었는가"를 잰다.

2.4 작성 양식 — AI에게 줄 정보 (5요소)

요소내용
① 팀의 제품/기능 목표한 문장 Product Goal
② 해결하려는 사용자 문제누가, 무엇 때문에 불편한가
③ 기대하는 사용자 행동/변화제품이 잘 됐을 때 사용자가 무엇을 다르게 할 것인가
④ 현재 측정 가능한 데이터/요소로그·센서·이벤트 중 무엇이 잡히는가
⑤ 팀이 생각한 초기 가설"이걸 재면 될 것 같다"라는 후보

AI가 주는 초안 결과물 (5요소)

  • 적절한 BM 후보 / 각 지표의 선정 이유 / 측정 방법 / 필요한 로그·데이터·기능·인터페이스 / 우선 검토 추천안

2.5 팀 검토 5질문 (시험 출제 0순위)

  1. 이 지표가 정말 제품/기능의 성공을 설명하는가?
  2. 사용자 가치와 연결되는가?
  3. 단지 측정하기 쉬운 지표는 아닌가? ← 함정 지표 경계
  4. 실제 서비스 환경에서 수집 가능한가?
  5. 더 적절한 대안은 없는가?
📝 암기: "성·가·쉬·수·대" — 성공 설명 / 가치 연결 / 쉬운 지표 함정 / 수집 가능 / 대안

2.6 최종 산출물 6항목

  1. Product Goal / ② 최종 BM / ③ 선정 이유 / ④ 측정 방법 / ⑤ 필요한 추가 데이터·기능 / ⑥ AI 초안에서 수정·보완한 내용 ← 평가의 핵심

2.7 좋은 답 vs 나쁜 답

❌ 나쁜 답 "DAU(일간 활성 사용자)를 본다"
→ 측정만 쉬움. 사용자가 정말 가치를 얻는지 설명 못함. 검토 3번에 걸림.
✅ 좋은 답 "우리 앱의 핵심 가치는 '딴짓 후 복귀'다. BM은 잔소리 개입 후 5분 이내 집중 세션 복귀율이며, 이는 사용자가 도구를 통해 실제로 원래 행동으로 돌아왔는지를 직접 측정한다."
⚠️ Common Misconceptions
오해 1: "DAU·페이지뷰·체류시간이 좋은 BM이다." → 틀림. 이들은 측정만 쉬운 함정 지표다. 사용자가 가치를 얻었는지 직접 설명 못 함. (예: 체류시간 길다고 만족도 높은 게 아님 — 헤매고 있을 수도)
오해 2: "BM은 모델 정확도와 비슷한 거다." → 틀림. BM은 사용자 행동, MM은 모델 예측 품질. 다른 차원.
오해 3: "BM은 한 개여야 한다." → 일부만 맞음. Primary BM은 1개가 좋지만, Supporting BM 여러 개로 보완. 단, Primary가 흐릿하면 안 됨.

우리 Zan Sol 적용

  • Primary BM: 잔소리 개입 후 5분 이내 복귀율. 5분 기준은 screenertia 연구의 주의 완전 전이 시간대.
  • Supporting: 일일 누적 집중 시간 / D7 리텐션 / 단계별 복귀율 / 문구별 복귀율
  • "세션 완료율"을 폐기한 이유: 우리는 목표 시간 없는 경과 시간 모델 → 함정 지표가 됨
🌱 다른 프로젝트에 적용 — 예시 2: "스터디 모임 매칭 앱"
  • ❌ 나쁜 BM: "회원가입 수" (수집 쉬움 함정)
  • ✅ 좋은 BM: "매칭된 스터디가 4주 이상 활동한 비율" (사용자 진짜 가치 = 지속되는 스터디)

🌱 다른 프로젝트에 적용 — 예시 3: "중고거래 신뢰도 평가 앱"
  • ❌ 나쁜 BM: "신뢰도 점수 표시 클릭 수"

3 Model Metric 도출 (실습 2)

지시서: AI_lecture.pdf 슬라이드 6·7 · 실습 1의 BM을 개선할 모델의 평가 기준

3.1 묻는 것

"실습 1의 Business Metric을 개선할 수 있는 Model Metric을 도출하라."

3.2 왜 이 과제가 필요한가

💡 The Why 모델을 만들면서 "이 모델이 잘 되고 있는지"를 잴 척도가 필요한데, 그 척도가 BM과 연결되지 않으면 모델 개선이 비즈니스 개선으로 이어지지 않는다. 예를 들어 추천 모델의 NDCG가 올라갔는데 클릭률은 그대로면 그 모델은 무가치. MM은 항상 "이 MM이 1단위 좋아지면 BM이 얼마 좋아지는가"의 인과 가설과 함께 정의되어야 한다.

3.3 핵심 개념 — BM vs MM

예시 (추천)클릭률 (CTR), 전환율NDCG, Precision@K, AUC

  3. 측정 가능하고 해석 가능한가?
  4. 데이터 제약과 운영 환경을 고려했는가?

3.6 좋은 답 vs 나쁜 답

❌ 나쁜 답 "분류 모델 정확도 95%를 목표로 한다"
→ 무엇을 분류하는지 / 왜 95%인지 / BM 연결 모두 없음
✅ 좋은 답 "BM(복귀율) 개선을 위해 '문구 추천 Top-1 hit rate'를 MM으로 본다. Bandit이 단계·맥락별 가장 복귀를 유도하는 문구를 선택했는지를 매 개입마다 측정. Top-1 hit rate가 1%p 오르면 평균 복귀율이 X%p 오른다는 가설을 A/B 테스트로 검증."
⚠️ Common Misconceptions
오해 1: "Accuracy 높으면 좋은 모델이다." → 틀림. 클래스 불균형 시 다수 클래스만 맞춰도 accuracy 높음. F1·Precision·Recall로 보완 + 무엇보다 BM과의 연결.
오해 2: "MM은 한 개로 충분." → 일부만 맞음. Primary MM 외에 운영 환경 메트릭(Latency·메모리 등)도 함께 본다.
오해 3: "MM은 모델 만든 뒤 정한다." → 틀림. MM 먼저 정해야 모델 선택이 가능.

우리 Zan Sol 적용

  • V1은 모델 없음 → MM도 없음. 운영 자체가 라벨 생성 (단계별·문구별 복귀율 누적).
  • V2 모델 ① 딴짓 예측: Binary Classification. MM = ROC-AUC + Precision@10min. BM 연결: 선제 개입으로 1·2·3차 진입 자체를 줄여 복귀율 ↑
  • V2 모델 ② 잔소리 개인화: Contextual Bandit (Thompson Sampling). MM = Cumulative Reward / Regret. BM 연결: 문구 hit rate ↑ → 단계별 복귀율 ↑
🌱 다른 프로젝트에 적용 — "교내 식당 대기 예측 앱"
  • BM: "예측 본 사용자의 만족도(NPS) + 잘못 갔다 다시 옮긴 비율 감소"
  • MM: "예측 오차 MAE 5분 이내" + "10분 이상 예측 어긋난 비율 < 5%"
  • BM 연결 가설: MAE 1분 줄어들면 만족도 X점 오른다 (A/B 테스트로 검증)

4 Freshness 검토

지시서: 2 In-Class-Assignment-Freshness.pdf · 2026-04-14

4.1 묻는 것

"우리 팀 프로젝트는 시간이 지나도 같은 성능을 유지할 수 있는가?"

4.2 왜 이 과제가 필요한가

💡 The Why ML 모델은 학습 시점의 현실을 반영한다. 시간이 흐르면 사용자 취향, 시장, 정책, 데이터 유형이 바뀌어 모델이 "낡는다(stale)". 이걸 안 보면 출시 후 성능이 서서히 죽는데도 모르고 운영하다가 사고가 난다. Freshness 검토는 "이 모델이 6개월 뒤에도 작동할까"를 미리 시뮬레이션하는 작업.

4.3 핵심 개념 — Freshness 정의

📌 정의 (지시서 원문) Freshness = 모델이 학습한 데이터가 시간이 지나도 여전히 현재의 현실을 잘 반영하는지, 그리고 데이터·환경의 변화로 인해 모델 성능이 저하될 가능성이 있는지.

현실이 바뀌는 4가지 이유

요인예시
① 사용자 취향 변화유행, 세대 차, 라이프스타일
② 계절/시기별 패턴 변화학기, 휴가, 명절, 행사
③ 시장·제도·정책 변화법 개정, OS 정책, 약관 변경
④ 신유형 데이터 등장신조어, 신규 앱, 신규 기기

4.4 핵심 6질문

  1. 데이터 변화 가능성 — 시간 지나면 달라지는가?
  2. 변화의 내용 — 무엇이, 왜, 어떻게?
  3. 성능 영향 — 모델 예측·신뢰도에 어떤 영향?
  4. 재학습 필요성 — 데이터 갱신/모델 재학습 필요?
  5. 점검 주기 — 얼마나 자주?
  6. 실행 부담 — 비용·시간·인력?

4.5 작성 양식 6항목

  1. 주제 / ② 사용 데이터 + 어떤 현실 / ③ Freshness 위험 요인 / ④ 예상 영향 / ⑤ 대응 방안 / ⑥ 실행 가능성

4.6 좋은 답 vs 나쁜 답 (지시서 직접 명시)

❌ 부족한 답 "데이터는 변할 수 있다"
✅ 좋은 답 "우리 프로젝트에서는 사용자 선호가 학기 초와 학기 말에 달라질 수 있으므로 학기 단위로 데이터를 점검할 필요가 있다"

→ "자기 팀 프로젝트 맥락 + 구체 수치/주기" 공식.

⚠️ Common Misconceptions
오해 1: "Freshness는 ML 모델만의 문제다." → 틀림. 규칙 기반 시스템도 룰이 낡는다 (예: 딴짓 앱 리스트). 모든 데이터 기반 의사결정에 적용.
오해 2: "재학습 자주 하면 해결된다." → 일부만 맞음. 재학습 자체에 비용이 큼. 점검 주기·자동화 비용·라벨 갱신 부담까지 함께 봐야.
오해 3: "데이터가 많이 쌓이면 자동으로 fresh해진다." → 틀림. 옛날 데이터가 많아도 분포가 바뀌면 오히려 노이즈.

  • 수요 예측: 계절·행사·이슈
  • 텍스트 분류: 신조어·신규 이슈
  • 이미지 분류: 촬영 환경·조명·장비

5 Speed 검토

지시서: 3 In-Class-Assignment-Speed.pdf · 2026-04-17

5.1 묻는 것

"우리 팀 프로젝트는 사용자가 기대하는 시간 안에 결과를 제공할 수 있는가?"

5.2 왜 이 과제가 필요한가

💡 The Why 모델이 정확해도 답이 5초 뒤에 나오면 사용자는 떠난다. Speed는 BM을 지원하는 필수 조건이며, 학생들은 보통 모델 추론 속도만 보다가 DB 조회·네트워크·기기 성능 등 다른 병목을 놓친다. 이 과제는 "전체 파이프라인의 누적 지연"을 보게 하는 훈련.

5.3 핵심 개념 — Speed의 진짜 정의 (가장 자주 틀림)

⚠️ 가장 자주 틀리는 정의 Speed ≠ 모델 추론 시간만!
Speed = 사용자가 요청을 보낸 뒤 결과를 받기까지 전체 대기 시간.

전체 대기 시간 구성 5요소 (시험 단답)

  1. 모델 추론 시간
  2. 데이터 불러오기 시간
  3. 데이터베이스 조회 시간
  4. 네트워크 지연 시간
  5. 여러 단계 처리 누적

5.4 왜 Speed가 BM을 지원하는가

  • 자동완성: 입력 중에도 빠르게 (반복 호출 누적)
  • 추천: 기다리기 전에 결과를 보여줘야 (이탈 방지)
  • 번역·진단·분류·검색: 늦어질수록 만족도 ↓

5.5 핵심 6질문

  1. 기대 대기 시간
  2. 최소 속도 요구조건 (minimum speed requirement)
  3. 예상 처리 시간
  4. 병목 요인 (모델 vs 데이터·DB·네트워크·기기)
  5. 전체 파이프라인 검토 (한 번 vs 누적)
  6. 대안 탐색

5.6 함께 고려할 5요소

요소설명
① 모델 복잡도복잡 → 처리 시간 ↑
② 데이터 크기·전처리크기/가공 복잡 → 시간 ↑
③ 반복 실행 여부한 번 vs 입력마다 (자동완성류)
④ 비모델 요인DB / 네트워크 / 서버 / 기기
⑤ 개발·유지보수성빠르더라도 개발 어렵면 비현실

5.7 좋은 답 vs 나쁜 답 (지시서 직접 명시)

❌ 부족한 답 "우리 프로젝트는 빨라야 한다"
✅ 좋은 답 "사용자가 버튼을 누른 뒤 2초 이내에 결과를 받아야 하며, 현재 구상은 이미지 업로드와 서버 추론 때문에 5초 이상 걸릴 가능성이 있으므로 이미지 크기 축소 또는 더 단순한 모델로 변경하는 방안을 검토해야 한다"

→ "수치 + 병목 지점 + 대안"의 3종 세트.

⚠️ Common Misconceptions
오해 1: "GPU 쓰면 빠르니 OK." → 틀림. GPU 추론이 50ms여도 네트워크 + 이미지 업로드가 3초면 사용자에겐 3초.
오해 2: "한 번 실행 시간만 측정하면 된다." → 틀림. 자동완성처럼 입력마다 호출되는 기능은 누적 시간이 진짜 병목.
오해 3: "단순한 모델은 좋지 않다." → 틀림. 단순 모델 = Speed의 강점. Zan Sol 사례 참조.

우리 Zan Sol 적용 — Speed 재프레임

"몇 ms"가 아니라 "도피 시작부터 몰입 전까지의 골든 타임 안에 개입이 성사되는가".

  • 최소 속도 요구: 1초 — Nielsen "사고 흐름 끊김 한계"
  • 구조적 강점 (3중 중첩): ① 이벤트 기반(폴링 0) ② 로컬 완결(네트워크 0) ③ 규칙 기반(모델 추론 0)
  • 단일 크리티컬 패스: Speed가 진짜 중요한 지점은 오직 하나 — AccessibilityEvent → 오버레이
  • 결론: "단순함 자체가 Speed의 가장 강력한 보장"
🌱 다른 프로젝트에 적용 — "이미지 기반 음식 칼로리 계산 앱"
  • 기대 대기: 사진 찍은 뒤 3초 이내 결과
  • 현재 예상: 사진 업로드 2초 + 서버 추론 1.5초 + DB 조회 0.3초 = 3.8초 (초과)
  • 병목: 이미지 업로드 (네트워크)
  • 대안: 클라이언트 이미지 압축 (300KB → 100KB) / 카메라 해상도 낮춤 / 모바일 추론 (서버 X)

6.1 묻는 것

"데이터가 부족할 때, 라벨링·기존 자원으로 어떻게 시작하는가?"

6.2 왜 이 과제가 필요한가

💡 The Why 대부분의 학생 프로젝트는 데이터가 부족한 채 시작한다. 처음부터 무에서 시작하면 학기를 다 쓰고도 못 끝낸다. 이 과제는 "라벨링으로 휴리스틱의 씨앗 만들고, 기존 자원으로 시작점 빌리고, 어디서 막힐지 미리 본다"는 3종 훈련. 특히 "애매했던 사례"를 적게 하는 것은 현실의 복잡성을 직시하라는 의도.

6.3 3 Part 구성

Part A. Labeling Examples

  • 5~10개 직접 라벨링
  • 각 예시마다 5요소:
  1. 라벨 이름 / ② 라벨링 이유 / ③ 판단 기준 / ④ 애매했던 사례 여부 / ⑤ 발견한 패턴
📝 암기: "이름·이유·기준·애매·패턴"
💡 왜 "애매했던 사례"를 적어야 하나 애매한 사례 = 규칙이 못 잡는 부분 = 모델이 필요한 진짜 지점. 명확히 분류 가능한 케이스는 규칙으로 충분, 애매한 케이스가 ML의 가치 영역. 시험에 "왜 애매한 사례를 기록하는가" 서술로 나올 수 있음.

Part B. Similar Public Implementations

  • 비슷한 모델 / 데이터셋 / 입출력 구조 / 파이프라인 / 문제 정의 중 하나 이상

7 도메인 전문가를 활용한 프로젝트 맥락 이해

지시서: 실습 0416.pdf · 2026-04-16 · 인클래스 15분

7.1 묻는 것

"이 문제를 사람이 어떻게 푸는가? 그 절차로 heuristic을 만들라."

7.2 왜 이 과제가 필요한가

💡 The Why ML로 풀기 전에 사람이 어떻게 푸는지 모르면, 모델이 무엇을 학습해야 하는지도 모른다. 도메인 전문가의 절차 = ML 모델의 시작점. 또한 그 절차에서 "사람의 판단이 들어가는 지점"이 곧 ML이 가치 있는 영역. 이 과제는 8-Step의 Step 6("알고리즘이 되어보기")의 외부 지식 확장판.

7.3 핵심 개념 — Heuristic

📌 정의 (지시서 원문) Heuristic = 입력이 들어왔을 때 여러 단계를 거쳐 결과에 이르는 과정 속에 사람의 판단이나 의사결정이 일부 포함된 절차적 해결 방식.

형태: "먼저 A 확인 → X면 B로 가고 → Y면 C 확인 → 최종 D 결정"의 3~5단계.

💡 왜 3~5단계인가 2단계 이하면 너무 단순해서 ML 가치를 못 보여줌. 6단계 이상이면 학기 안에 모두 모델링 못 함. 3~5가 "사람의 판단을 옮길 만큼 풍부 + 학기 안에 다 다룰 수 있는" 골디락스 존.

7.4 작성 양식 6항목

항목적어야 할 내용
① 팀 프로젝트 주제한 줄 정의
② 도메인 전문가 후보누가 잘 푸는가 (이론·실전·역방향 그룹 권장)
③ 얻고 싶은 지식현재 어떻게 푸는가 / 판단 기준 / 예외 / 제약
④ 지식 확보 방법인터뷰 / 웹 / 도서 / 인터뷰 기사·사례
⑤ 간단한 heuristic3~5단계. 사람 판단 개입 지점 명시
⑥ scope/challenge일반화 한계 / 변수 / 접근 가능성

7.5 좋은 답안의 특징

  • 전문가 후보가 구체적 (역할·기관·접근 방법)
  • 얻고 싶은 지식이 판단 기준 + 예외 + 제약까지
  • Heuristic이 사람의 판단 개입 지점 표시
  • scope·challenge에 "이게 ML이 필요한 이유"까지 연결
⚠️ Common Misconceptions
오해 1: "전문가는 그 분야 박사·교수만 가능." → 틀림. 현장 실무자(교사·간호사·운전기사 등)가 더 좋을 수도. 잘 푸는 사람이면 됨.
오해 2: "Heuristic은 ML이 만들기 전 임시방편." → 틀림. Heuristic은 V1에서 그대로 운영되고 데이터를 쌓는다. ML로 가는 과도기가 아니라 독립적 가치.
오해 3: "전문가 인터뷰 못 하면 끝." → 틀림. 인터뷰 기사·도서·"이 사람이라면 어떻게 풀까" 상상까지 인정 (지시서 명시).

우리 Zan Sol 적용 — 3그룹 인터뷰 + 3단계 Heuristic

그룹주는 지식
① 학습상담사이론 — 어떤 순서로 개입
② 자습감독 교사실전 — 어떤 말이 먹히는가
③ 4.0+ 선배역방향 — 도피 안 하는 사람의 규칙

핵심 단일 질문: "도피 중인 학생에게 개입했을 때, 실제로 돌아오게 만든 경험이 있는가? 그때 무엇이 먹혔는가?"

Heuristic (3단계): 즉시 행동제안 → 30초 공감 → 2분 압박. 사람 판단 개입 = "어떤 스타일이 먹히는가" 선택. V2 Bandit으로 자동화.

🌱 다른 프로젝트에 적용 — "교내 행사 추천 앱"
  • 전문가: ① 학생회 임원 (이론) ② 동아리 회장 (실전) ③ 행사 잘 챙기는 선배 (역방향)
  • 핵심 질문: "어떤 행사가 학생들에게 진짜 도움이 됐고, 어떤 행사가 외면받았나?"
  • Heuristic: 학과 → 학년 → 관심 키워드 → 시기 → 추천 (5단계). 판단 개입 = 키워드 가중치
  • Challenge: 학생마다 키워드 기준이 다름 → V2에서 협업 필터링

🎯 시험 출제 가능

  • Heuristic 정의 — 단답
  • "왜 ML 모델 설계 전에 도메인 전문가가 필요한가" — 서술
  • 전문가 지식 확보 방법 4가지 — 단답
  • 사례 주고 "3~5단계 heuristic 작성" — 적용
  • "사람이 푸는 절차"와 ML이 필요한 지점의 관계 — 서술

8 UX 설계 — IA · User Flow · Wireframe

지시서: 0428_UX설계_안내문.pdf · 2026-04-28

8.1 묻는 것

"개발 직전, 서비스의 구조 · 사용자 흐름 · 화면 구성을 명시화하라."

8.2 왜 이 과제가 필요한가

💡 The Why 좋은 모델·BM·Heuristic을 다 만들어도 사용자가 그 기능을 못 찾으면 무가치. UX 설계는 "여기까지의 모든 결정이 사용자에게 어떻게 보이는가"를 강제로 그려보게 하는 단계. 또 4인 팀이 16주 안에 무엇을 누가 만들지 분담 기준도 여기서 나온다.

  3. 입력 또는 확인 단계
  4. 의사결정 지점
  5. 예외 상황 1개 이상
  6. 완료 화면

핵심 메시지

8 Step 전체

Step 1 — Product Goal 먼저 정의

좋은 목표의 조건 3가지:
  • 사용자 중심으로 설명 가능
  • 결과물이 명확
  • "무엇을 자동화/추천/예측/생성할 것인지" 드러남

Step 2 — Input/Output으로 다시 쓰기

  • Input·Output 정의 / 단위 / Pair 확보 가능 여부
  • 예: 번역 → Input 한국어, Output 영어 / 추천 → Input 검색어+행동, Output 클릭 가능 아이템

Step 3 — Data 4 수준 (시험 0순위)

수준정의예시
① Ground Truth / LabeledI/O 정확히 연결가장 유용 / 가장 드뭄
② Weakly Labeled완벽 정답 X, 간접 신호클릭 / 좋아요 / 체류 / 평점
③ Unlabeledoutput label 없음라벨링 / 규칙 설계 필요
④ No Data아직 없음수집 계획부터
📝 암기: "GT · Weakly · Unlabeled · None"

Step 4 — Generative Model 체크

체크 질문 3가지:
  • 출력이 자유롭게 생성되어도 되는가?
  • 일관성과 통제가 더 중요한가?
  • 잘못 생성됐을 때 리스크가 큰가?
가이드: 생성형 쓰려면 (1) 왜 꼭 필요한지 (2) 비생성형 대안도 함께

Step 5 — End-to-End vs Simple Solution

End-to-EndSimple Solution
입력 → 출력 한 번에 생성. 직관적·멋져 보이지만:
  • paired data 많이 필요
  • 복잡도·학습 부담 ↑
  • 디버깅 어려움
작은 단계로 나눔:
  • 구하기 쉬운 데이터
  • 해석·구현 쉬움
  • 속도·유지보수 유리
원칙: "가장 화려한 모델" X "이번 학기 구현 가능한 모델" O

Step 6 — '알고리즘이 되어보기'

복잡 모델 고르기 전, 사람이 직접 풀 때 어떤 기준 쓸지 먼저 생각 → 필요한 feature 발굴.

Step 7 — Feature 기반 단순 모델

점수화 / 분류 / 순위화부터. 첫 버전은 분류·점수·랭킹·추천이 더 현실적.

Step 8 — 데이터 부족 시 framing 변경

예시 진화: "사용자가 더 나은 질문 작성 돕기" → "나쁜 질문 → 좋은 질문 자동 생성" → "질문 quality score 예측" / "좋은/나쁜 질문 분류" / "개선 필요 feature 추천"
3가지 원칙:
  1. 데이터 부족 → 모델 바꾸고
  2. 모델이 너무 무거우면 → 문제를 다시 정의
  3. 프로젝트 설계는 반복적(iterative) 과정

좋은 실습 결과의 기준 (수업 마무리)

  • 주제가 흥미로운가보다 데이터·모델이 연결되어 있는가
  • I/O 명확 / 데이터 확보 계획 현실적 / 생성형 사용 정당 / 대안 framing 비교 / 학기 안 구현 가능

검토 5질문 적용:

Q2-1. 이 앱의 Freshness 위험 요인 2가지와 각각의 대응 방안을 서술하라. (서술 5점)
✅ 모범 답안 위험 요인 1: 계절·시기별 재료 변화
- 계절에 따라 시장에 나오는 재료가 다르고 사용자의 선호 메뉴도 변함 (여름엔 냉면, 겨울엔 국밥). 학습 데이터가 한 계절에만 치우치면 다른 계절 추천 부정확.
- 대응: 계절별 데이터를 분기별로 재학습 + 추천 시 현재 월 가중치 부여. 신메뉴는 사용자 평점 기반 점진적 도입.

위험 요인 2: 신규 식재료/유행 메뉴 등장
- "곤약파스타", "두바이 초콜릿" 같은 유행 메뉴는 학습 시점에 없던 데이터.
- 대응: 매월 SNS·검색 트렌드 모니터링 + 신규 키워드 자동 라벨링 큐 + 사용자 신고 기능으로 누락된 재료 보고받기.
채점 포인트: 자기 프로젝트 맥락 / 구체 수치(분기별·매월) / "왜 문제 + 어떻게 대응" 함께 / 일반론 회피
📋 시나리오 3. "교내 행사 추천 앱" 팀. 학과·학년·관심사 기반으로 행사를 추천한다. 데이터는 No Data 상태에서 시작.
Q3-1. 이 팀의 Data·Model 8-Step 중 Step 3(Data 4 수준)와 Step 8(framing 변경)을 어떻게 적용할 수 있는가? (서술 7점)
✅ 모범 답안 Step 3 (Data 4 수준):
- 현재 ④ No Data. 학생들이 어떤 행사에 관심 있는지 기록 없음.
- V1 운영을 통해 ② Weakly Labeled로 진화 가능: 행사 클릭 / 즐겨찾기 / 신청 클릭 등 간접 신호 수집.
- 누적 사용자 200명·행사 100개 이상 시 ① Labeled: "이 학생이 이 행사에 만족" pair 확보.

Step 8 (framing 변경):
- 원래 목표: "사용자에게 가장 만족할 행사를 자동 추천" (paired data 필요 → 학기 안 불가)
- Framing 변경: "사용자가 행사를 거를 수 있도록 학과·학년·관심사 필터링 + 인기 순 정렬" — 추천 ML 대신 규칙 기반 필터 + 클릭 카운트 정렬로 V1 framing.
- 결과: 데이터가 부족해도 V1 출시 가능 + 운영 중 데이터 쌓이면 V2에서 협업 필터링으로 확장.
채점 포인트: Data 4 수준 정확 / 진화 경로 명시 / Step 8의 "데이터 부족 → framing 변경" 핵심 적용 / V1/V2 분리 인용
📋 시나리오 4. "주차장 빈자리 예측 앱" 팀이 사용자 입력 시 주변 주차장 빈자리 수를 5초 안에 보여주려 한다. 현재 구상: 사용자 위치 → 서버 → 각 주차장 API 조회 → 평균 빈자리 계산 → 응답.
Q4-1. 이 앱의 Speed 검토를 핵심 6질문 중 4개 이상 적용하여 서술하라. 병목 요인과 대안을 반드시 포함. (서술 7점)
✅ 모범 답안 ① 기대 대기 시간: 사용자가 주차장 찾는 상황은 운전 중 → 2초 이내가 현실적 기대.
② 최소 속도 요구조건: 5초 넘으면 사용자가 직접 차로 가는 게 빠름 → 5초가 절대 상한.
③ 예상 처리 시간: GPS 응답 1초 + 서버 호출 0.5초 + 5개 주차장 API 병렬 조회 (각 1~3초, 가장 느린 게 결정) ≈ 4~5초. 기대치 초과 위험.
④ 병목 요인: 주차장 API 응답 시간이 가장 큼. 일부 주차장은 자체 서버 노후로 3초까지. 모델·DB·네트워크는 부차적.
⑥ 대안: (a) 사전 캐싱 — 5분마다 API를 미리 호출해 DB에 저장, 사용자 요청 시 DB만 조회 (1초 이내). 단점: 5분 오차 허용. (b) 응답 느린 주차장은 별도 표시 없이 우선 빠른 것부터 표시.
최종 판단: Product Goal("운전 중 빠른 안내")을 위해 사전 캐싱이 정답. 5분 오차는 운전 중 의사결정에 무시할 수준.
채점 포인트: 6질문 중 4+ 명시적 적용 / 수치(2초·5초·4~5초) / 병목 정확 식별(API) / 대안 2개+ / Product Goal 연결
