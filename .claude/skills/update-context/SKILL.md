# update-context — 프로젝트 맥락 갱신 스킬

Tom이 새 정보를 주면 실행한다. 대충 아무 데나 덧붙이지 말고, **비교 → 분류 → 제안 → (질문) → 반영 → 기록** 단계를 지킨다.

---

## Step 1. 정보 수령 & 이해

Tom이 주는 정보는 어떤 형태든 될 수 있다. 예:
- 교수 피드백 한 문장
- 팀 회의 결과 메모
- 카톡 발췌
- 새 레퍼런스(논문·사례) 제시
- "V1 스코프에서 X를 빼자" 같은 결정
- 잔소리 문구 4축 외 새로운 축 아이디어
- 용어 정의 변경
- 서베이 n 추가 수집 결과

**먼저 1~2줄로 내가 이해한 바를 Tom에게 되비춰 확인한다.** 이해가 틀린 상태로 저장 금지.

---

## Step 2. 정보 유형 분류

다음 중 어디에 가까운지 판별한다 (복수 가능):

| 유형 | 기본 저장 위치 |
| --- | --- |
| (A) 확정된 새 의사결정 | `CONTEXT.md` Section 5 (신규 5-N 번호) |
| (B) 아직 확정 안 된 논의거리 | `CONTEXT.md` Section 1 (TEAM DISCUSSION QUEUE) |
| (C) 기존 결정의 **수정/보강** | 해당 섹션 내 수정 (Section 5-N 중 하나) |
| (D) 교수 피드백·과제 결과 교훈 | `MEMORY.md` 날짜 헤더 신규 entry |
| (E) 수치·서베이 결과·레퍼런스 | `CONTEXT.md` Section 11/16 또는 해당 섹션 |
| (F) 용어 정의 신규/변경 | `CONTEXT.md` Section 9 |
| (G) 스코프 변경 (In/Out Scope) | `CONTEXT.md` Section 8 |
| (H) 역할 분담 변경 | `CONTEXT.md` Section 15 |
| (I) 수업 프레임워크·포맷 업데이트 | `course/` 해당 파일 |
| (J) 팀 커뮤니케이션 패턴 관찰 | `MEMORY.md` |
| (K) 불확실 / 분류 애매 | **Tom에게 질문** |

---

## Step 3. 기존 내용과 비교 (Compare)

반영 전 다음을 **항상** 확인한다:

1. **중복:** 이미 같은 내용이 CONTEXT/MEMORY에 있는가? 있으면 덮어쓸지/보강할지/따로 추가할지 판단.
2. **충돌:** 기존 Section 5 결정과 모순되는가? 모순되면 Tom에게 **명시적으로 충돌을 보고**. 예: "이 정보는 5-3과 충돌합니다. 5-3이 여전히 유효한가요, 아니면 이 새 결정이 대체인가요?"
3. **근거 일관성:** 수치(30초, 500세션, 85.3% 등)가 기존과 다르면 반드시 **수치의 출처**를 물어본다. CONTEXT의 수치는 모두 근거가 있으므로 임의 수정 금지.
4. **용어 일관성:** Section 9 용어 사전에 정의된 단어를 다른 의미로 쓰려 하면 flag.
5. **V1/V2 경계:** V1에 관한 정보인지 V2에 관한 정보인지 명확히 구분. 섞이면 질문.

---

## Step 4. 업데이트 계획 제시 (Propose)

다음 양식으로 Tom에게 제시한다. 즉시 파일 수정하지 않는다.

```
### 업데이트 계획

**정보 요약:** <1줄>
**분류:** <A~K 중 어느 유형>
**영향받는 파일/섹션:**
- CONTEXT.md §<번호> <제목> — <추가 / 수정 / 이동>
- MEMORY.md — <날짜 entry 추가>
- course/<file>.md — <변경 내용>

**비교 결과:**
- 중복: <있음/없음, 있으면 어떤 항목>
- 충돌: <있음/없음, 있으면 어느 항목과>
- 근거/수치: <OK / 불명확 → 질문 필요>

**제안하는 변경 (diff 요지):**
- <기존>: ...
- <변경 후>: ...

**Tom 확인이 필요한 질문:** (있으면)
1. ...
2. ...
```

---

## Step 5. Tom의 결정 받기

Tom 응답 유형별 대응:
- **"승인"** → Step 6 진행
- **"X 부분만 수정해"** → 계획 고쳐 다시 제시 (Step 4 재실행)
- **"질문에 답변: ..."** → 답변 반영 후 계획 업데이트
- **"보류"** → 반영하지 않고, 필요하면 Section 1 TEAM DISCUSSION QUEUE에 "보류된 항목" 한 줄만 기록 (Tom 승인 후)

---

## Step 6. 반영 (Apply)

승인된 계획대로 파일 수정. 수정 시 규칙:

- `CONTEXT.md` Section 5 신규 결정 추가 시: `DECISION / WHY / COURSE LINK / CONTEXT` 4부 구조 유지.
- Section 번호 (5-1, 5-2, ...)는 **다음 번호** 부여. 중간 삽입으로 번호 재정렬은 Tom 승인 없으면 금지.
- 기존 섹션 수정 시: 수정 후 일관성 체크 (예: 5-3을 수정했으면 Section 14 메타 레슨에서 5-3 언급하는 부분 확인).
- 중요한 결정/피드백은 **MEMORY.md에도 한 줄** 추가 ("YYYY-MM-DD — 제목" 헤더 + 3 bullet 이하).
- CONTEXT.md Section 17 (문서 메타) 업데이트는 필요 시에만.

---

## Step 7. 결과 보고

Tom에게 다음을 1~2줄로 보고:
- 무엇을 어디에 반영했는지
- MEMORY.md에 추가한 한 줄 (있으면)
- 다음 단계 (예: "팀에 공유 필요하면 /kakao-team으로 초안 드릴까요?")

## 실제 사용 예시

### 예시 1: 교수 피드백

**Tom 입력:**
> "교수님이 '서베이 n=34는 절대 정량적으로 해석하지 마라, 탐색적으로만 써라'고 했어"

**내 응답:**
```
정보 요약: 교수 피드백 — 서베이 n=34는 탐색적으로만 해석하라
분류: (D) 교수 피드백 + (C) 기존 내용 보강

영향:
- CONTEXT.md §11-3 "표본 한계" 이미 존재. 문구 강화만 필요.
- MEMORY.md 2026-04-21 entry 추가 (향후 과제에서 반복 주의)

비교:
- 중복: §11-3에 유사 주의 있음 ("대표성 주장보다 방향성 확인 수준")
- 충돌: 없음
- 강화 제안: "정량 해석 금지"를 명시 추가

변경 diff:
기존: "대표성 주장보다 방향성 확인 수준으로 해석."
변경: "대표성 주장 금지, **정량 해석 금지**. 탐색적 조사로만 해석."

질문: 없음. 바로 반영해도 될까요?
```

### 예시 2: 새 스코프 결정

**Tom 입력:**
> "V1에 TODO 자동 임포트 기능 넣자, 구글 캘린더에서"

**내 응답:**
```
정보 요약: Google Calendar → TODO 자동 임포트를 V1 스코프에 추가 제안
분류: (G) 스코프 변경 + (A) 신규 결정 잠재

영향:
- CONTEXT.md §8-1 In Scope 추가 (P? 우선순위)
- CONTEXT.md §5 신규 결정 5-N 후보
- §5-8 아키텍처에 Google OAuth 포함 여부 재검토 필요

비교:
- 충돌: §8-1 현재 "P0 TODO/마감일 등록"에 "수동 입력" 전제. 이 신규 기능은 자동화 추가.
- §3-3 개발 기간 16주, 4인 팀 제약 → 우선순위 부여 필요
- V1/V2 경계: 이게 V1인지 V2인지 명확히 해야 함

질문:
1. 우선순위는 P0/P1/P2 중 어느 쪽?
2. 이번 학기 16주 내에 넣는 게 맞는지, V2로 미뤄야 하는지?
3. Google OAuth 추가는 아키텍처 §5-8에 영향. 수용?

일단 결정하기 전에 Section 1 TEAM DISCUSSION QUEUE에 올려서 팀 논의부터 할까요?
```

### 예시 3: 작은 수치 보강

**Tom 입력:**
> "설문 프리미엄 지불의향 35%였는데 재집계하니 31%야"

**내 응답:**
```
정보 요약: 서베이 프리미엄 지불 의향 수치 35% → 31% 정정
분류: (E) 수치 업데이트

영향: CONTEXT.md §11-1 표 수정 한 군데

비교:
- §11-1 표에 "~35%" 기재됨
- 근거 변경 있나요? 재집계 방식 동일한 대상에서 나온 결과인가요? (수치 수정은 근거 확인 필수)

질문:
- 재집계가 같은 n=34 응답에서 다시 뽑은 건가요, 아니면 추가 응답 포함인가요?
- 이 차이가 답안에 유의미한가요, 아니면 "~30%대" 수준으로 퉁쳐도 되는가요?
```

## 하지 말아야 할 것

- 비교/분류 없이 바로 파일 수정
- Section 5 결정 번호를 건너뛰거나 재정렬
- 수치/근거 변경을 질문 없이 반영
- 불확실한 분류를 임의 결정
- MEMORY.md에 2줄 이상 장황하게 적기 (짧게 압축)
- CONTEXT.md가 300줄 넘게 비대해지도록 방치 (섹션별 중복 제거)
