1. Information Architecture

1-1. 주요 사용자 유형

V1은 단일 유형: 개인 사용자(학생 본인). 팀/관찰자/관리자 역할 없음.

1-3. 설계 의도

  • 홈 → 세션이 1탭 안에 닿음. 핵심 루프가 가장 짧은 경로.
  • 리포트·설정은 보조 메뉴로 분리 (사용 빈도 낮음).
  • 잔소리 오버레이는 메뉴 외부에서 트리거됨 — 시스템 권한(AccessibilityService + SYSTEM_ALERT_WINDOW)으로 다른 앱(인스타·유튜브) 위에 그려진다. 트리상 "세션" 하위지만 실제로는 OS 레이어 위에 떠 있다.
  • 비슷한 기능 중복 없음: TODO 등록은 "할 일"에만, 권한 관리는 "설정"에만.

2. User Flow

2-1. 선택한 핵심 사용자 목표

"집중 세션을 시작하고, 딴짓을 시도했을 때 잔소리 개입을 받아 원래 세션으로 복귀한다."

이 목표를 고른 이유: 우리 서비스의 Product Goal("그 순간 즉시 개입하여 복귀") 자체가 이 흐름이다. 다른 후보(온보딩, TODO 등록, 리포트 보기)는 보조 흐름.

2-3. 의사결정 지점

  1. AccessibilityService 권한 부여 여부
  2. 실행 앱이 사용자가 등록한 딴짓 앱 리스트에 있는가
  3. 1차 잔소리 후 30초 내 복귀했는가
  4. 2차 잔소리 후 2분 내 복귀했는가
  5. 3차 잔소리 후에도 무시하는가

2-4. 예외 상황

  • 예외 1: 권한 미부여 — AccessibilityService 권한이 없으면 핵심 루프 자체가 작동 불가. 첫 세션 시작 시 권한 안내 화면으로 강제 분기.
  • 예외 2: 3단계 모두 무시 — 사용자가 끝까지 안 돌아오는 케이스. 세션을 "실패"로 기록하고 리포트에 명시 (다음 세션 설계 입력값).

Wire Flow 점검
  • 화면 간 이동 자연스러움: 핵심 루프가 1탭 깊이.
  • 다음 행동 제시: 모든 종료 화면(리포트)에 [다음 세션] CTA.
  • 불필요 이동 없음: 세션 진행 중 잔소리는 별도 화면이 아니라 시스템 오버레이.
  • 예외 처리: 권한 미부여 → 설정 분기 / 끝까지 무시 → 실패 리포트.

5. 설명 문서

  1. 핵심 사용자는 관찰자 없이 혼자 공부하다가 스마트폰으로 도피하는 경희대 대학생이며, 자체 서베이(n=34)에서 도피 경험 85.3%로 문제가 충분히 크다.
  2. 핵심 문제는 도피 "그 순간"에 개입할 수단이 없다는 것이다 — 기존 앱(스크린타임은 사후 리포트, Forest는 의지 전제)은 이 순간을 비워둔다.
  3. IA는 "홈 → 세션"이 1탭 안에 닿도록 핵심 루프를 가장 짧은 경로로 배치하고, 리포트·설정은 보조 메뉴로 분리하여 사용자가 매번 의식적으로 찾을 필요가 없게 했다.
  4. User Flow에서 가장 중요하게 고려한 행동은 "딴짓 앱 실행 → 1차/2차/3차 잔소리 → 복귀"의 단일 핵심 루프이며, 세 번의 의사결정 지점(각 단계 무시 여부)이 우리의 Primary Metric인 복귀율의 직접 입력값이 된다.
  5. Wireframe에서 가장 중요하게 배치한 것은 잔소리 오버레이의 [돌아가기] 버튼을 가장 큰 CTA로 두고 1·2차에는 닫기를 두되 3차에는 닫기를 제거한 것이다 — 이는 행동활성화 → 자기효능감(Bandura) → 심리적 저항 회피(Reactance)의 단계별 심리 메커니즘을 화면 구조로 옮긴 것이다.
  6. 이 설계는 개발 전에 핵심 루프가 디바이스 로컬에서 완결됨을 시각적으로 확인시켜 주고(네트워크 지연이 Product Goal에 영향 없음), 4인 팀이 16주 안에 어떤 화면을 누가 만들지 분담 기준을 제공한다.
